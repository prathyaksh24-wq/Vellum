# Memory and Knowledge Ownership

Vellum exposes one memory system and one maintained knowledge wiki. The
storage engines behind them are implementation details, not separate products.

## Canonical ownership

| Concern | Canonical owner | Supporting adapters |
| --- | --- | --- |
| Chat history | `data/ui/conversations.json` through `/api/conversations` | Obsidian conversation projection, FTS5 index |
| Durable memories, status, settings, summaries | Memory Orchestrator SQLite store | FTS5, Chroma memory cards, Honcho, optional provider extensions |
| User-readable conversation archive | `Vault/Agent/Conversations` | Generated from canonical chat history |
| User-readable memory snapshots | `data/memory/USER.md`, `data/memory/MEMORY.md`, and structured Obsidian cards | Generated by Memory Orchestrator/Dreaming |
| Maintained reference knowledge | `Vault/Knowledge` through Knowledge Wiki | Index, opaque page refs, version history, lint log |
| Raw imports | `Vault/Library` | Immutable and untrusted until the user explicitly approves an ingestion |

The frontend calls only stable `/api/memory/*`, `/api/conversations/*`, and
`/api/knowledge/*` contracts. It must not choose between SQLite, FTS5, Chroma,
Honcho, or provider extensions.

## Retrieval flow

1. The process-wide Memory Orchestrator builds the bounded memory packet.
2. SQLite supplies summaries and saved memories; FTS5 supplies relevant past
   turns; Honcho and enabled extensions may add context.
3. The Knowledge Wiki contributes only lightweight page references. Page bodies
   are read with `knowledge_wiki` on demand, so knowledge is not duplicated into
   personal memory.
4. Current conversation context always wins when older memory conflicts.

## Write flow

1. `/api/conversations` saves the canonical thread.
2. The background learner indexes the turn and extracts pending durable memory.
3. Dreaming promotes, merges, summarizes, archives, and audits memory changes.
4. The Obsidian exporter updates one human-readable note per conversation.
5. Knowledge pages change only through explicit wiki upsert/approved ingestion.

Legacy `Agent/Queries` and `Agent/Responses` files are migration inputs, not
canonical memory. New runtime behavior must not create timestamped per-turn
notes in those folders.
